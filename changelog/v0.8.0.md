# v0.8.0

Server discovery moved from MCP tool to CLI.

## Why

The `list_servers` MCP tool was a pure config read with a TCP probe — it never used SSH sessions or the connection pool. Having it as an MCP tool meant its JSON schema (~135 tokens) was loaded into every API call's context, and agents sometimes confused it with remote tools (e.g., calling `remote_bash` with server `"list"` instead of running the CLI).

The broader trend (Playwright CLI, GitHub CLI over GitHub MCP) confirms that discovery and management belong in the CLI layer, while MCP tools should focus on remote operations that benefit from structured parameters and stateful connections. For ssh-hub specifically: the tool count is small (~8 tools), so the token savings are modest, but the architectural clarity and LLM-friendliness improvements are significant.

## Changes

### Server discovery via CLI

- **`list_servers` MCP tool removed** — server discovery is no longer an MCP tool. Agents run `ssh-hub list` as a local shell command instead, following the same pattern as `gh` for GitHub.

- **`ssh-hub list` now probes reachability by default** — runs concurrent TCP probes against each server's SSH port (3s timeout, all servers probed in parallel). Shows `ok reachable (45ms)` or `warn unreachable` alongside each server's config and metadata. Use `--no-probe` to skip probing.

- **MCP instructions updated** — the server instructions now explicitly direct agents to run `ssh-hub list` as a LOCAL shell command (via Bash tool), not through any MCP tool. This eliminates the confusion where agents tried to discover servers through remote tools.

### LLM-friendly CLI output

- **All CLI output works in non-TTY environments** — `indicatif` spinners are invisible when stdout isn't a terminal (e.g., when an agent runs commands via Bash). Audited all CLI commands to ensure final status output always goes through `println!`, not through indicatif's rendering pipeline. The spinner provides visual feedback for humans in TTY mode; agents get the same information without it.

- **Error details surfaced in `ssh-hub add`** — connection failures now show the actual error (`Connection failed: host key mismatch`) instead of the generic `"failed authentication"` message. Metadata extraction errors also include the reason in the printed output instead of hiding it in debug logs.
